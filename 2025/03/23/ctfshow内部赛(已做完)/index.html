解释一下为什么是这个payload

在题目中会获取get传入的S赋值给变量S，如果变量S非空则执行eval语句

在eval语句中有$$S，内部的就是变量S，结果就是

第一个payload相当于先执行了system语句然后把结果赋值给变量a，然后eval将变量a的结果返回

第二个payload就是利用分号逐步执行eval语句的作用，$a是空变量所以啥都不会干，接着就会执行system语句

走不通，另外看看

想起了题目中说漏洞每两分钟触发一次，猜测可能是定时任务，查看一下定时任务

分析一下

  • 前五行是注释，说明了这个文件的用途和特性。

  • 接下来设置了 SHELL=/bin/sh 和 PATH 环境变量。

  • 然后是四个标准的系统 cron 任务，用于执行 hourly、daily、weekly 和 monthly 的定时任务。

  • 最后一个*/1 * * * * root /usr/sbin/logrotate -vf /etc/logrotate.d/nginx 是自定义的任务

这个任务是每分钟以 root 用户身份运行一次 logrotate 来处理 nginx 的日志。

发生的错误 /bin/sh: -c: line 1: syntax error near unexpected token ;’` 表明在执行某个 shell 命令时遇到了语法错误。这个错误不是来自 crontab 文件本身，而是来自其中一个被执行的命令或脚本。

从错误信息 **/bin/sh: -c: line 1: `;echo ‘**来看，该命令中包含了一个意外的分号;，接着是一个未闭合的引号。这通常是因为命令语法错误或者字符转义问题导致的。

产生原因：Nginx在创建log目录时使用了不安全的权限配置，可造成本地权限的提升，攻击手法主要通过从www-data权限向root用户权限提权得到。

#CVE-2016-1247

漏洞利用分析

一般来说，如果想要修改函数的功能，最直接的就是对其源码进行更改，但很多情况下我们是无法达成此目标的，这时就可以借助一些hook操作来改变程序的流程，从而实现对函数的修改。在 Linux 系统下，我们可以通过编译一个含相同函数定义的 so 文件并借助/etc/ld.so.preload文件来完成此操作，系统的 loader 代码中会检查是否存在/etc/ld.so.preload 文件，如果存在那么就会加载其中列出的所有 so 文件，它能够实现与 LD_PRELOAD 环境变量相同的功能且限制更少，以此来调用我们定义的函数而非原函数。此方法适用于用户空间的so文件劫持，类似于 Windows 下的 DLL 劫持技术。更进一步，如果我们将此技巧与含有suid的文件结合起来，那么就可以很自然的实现提权操作了，所给的 PoC 就是利用的这个技巧。
